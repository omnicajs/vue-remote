## Quﾃｩ cubre esta guﾃｭa

Esta guﾃｭa te ayuda a arrancar la capa de sincronizaciﾃｳn de render entre:

- un producto host que controla los componentes de UI permitidos;
- un runtime remoto independiente para extensiones.

`@omnicajs/vue-remote` se encarga del renderizado sincronizado.
Las APIs de negocio, la auth, el discovery y la lﾃｳgica de marketplace permanecen en tooling adyacente.

## Tooling para IDE

Para que las template refs nativas del lado remoto se tipen como proxies de elementos remotos en `*.remote.vue`, aﾃｱade el plugin de tooling de Vue:

```json
{
  "vueCompilerOptions": {
    "plugins": [
      "@omnicajs/vue-remote/tooling"
    ]
  }
}
```

Quﾃｩ esperar:

- `<div ref="panel" />` dentro de `*.remote.vue` se infiere como un proxy remoto de `@omnicajs/vue-remote/remote`, no como `HTMLDivElement`.
- `useTemplateRef(...)`, `ref(null)` y `shallowRef(null)` siguen el mismo modelo de frontera remota para refs nativas.
- Las refs a componentes host conservan el tipado de instancia de componente.

Recomendaciones para IDE:

- VS Code: usa la extensiﾃｳn `Vue - Official`.
- PhpStorm / WebStorm: usa `Vue Language Server (Volar)` y habilita el motor de tipos basado en servicios para obtener la evaluaciﾃｳn mﾃ｡s precisa.
- Otros IDE: si estﾃ｡n basados en `@vue/language-server` u otro cliente Vue compatible con Volar, tambiﾃｩn deberﾃｭan respetar `vueCompilerOptions.plugins`.

Si estﾃ｡s desarrollando dentro de este repositorio en lugar de consumir el paquete publicado, usa `./tooling.cjs` en `vueCompilerOptions.plugins`.

## 1. Inicializar el runtime host

En el lado host:

1. Registra los componentes host con `createProvider(...)`.
2. Crea el receiver del canal de render con `createReceiver()`.
3. Renderiza `HostedTree` y pﾃ｡sale `provider + receiver`.
4. Inicia el runtime remoto (iframe/worker) y pasa `receiver.receive` a `run(...)`.

Consulta: [Componentes del host](/es/host-components/) y [Resumen](/es/overview/).

## 2. Inicializar el runtime remoto

En el lado remoto:

1. Crea la raﾃｭz remota con `createRemoteRoot(channel, options)`.
2. Crea la app con `createRemoteRenderer(root).createApp(...)`.
3. Define los componentes expuestos por el host mediante `defineRemoteComponent(...)`.
4. Monta la raﾃｭz y la app dentro de `run(...)` y limpia en `release()`.

Consulta: [Componentes remotos](/es/remote-components/) y [Resumen](/es/overview/).

## 3. Conectar host y remoto

Usa un transporte por canal, normalmente `@remote-ui/rpc`, y mantﾃｩn un handshake estable:

1. El host carga la URL del runtime remoto.
2. El remoto expone `run/release`.
3. El host llama a `run(channel, hostBridge)`.
4. El remoto monta la UI a travﾃｩs del canal.
5. La sesiﾃｳn termina con `release()`.

Si prefieres un worker dedicado en lugar de transporte por iframe, consulta [Runtime de Web Worker](/es/web-workers/).
Para una configuraciﾃｳn especﾃｭfica de iframe, consulta [Integraciﾃｳn con iframe](/es/integration/).
Para handshakes explﾃｭcitos de canal entre contenedores personalizados, consulta [Transporte MessagePort](/es/message-port-transport/).
Para runtimes separados por proceso en Electron/Tauri, consulta [Transporte IPC de escritorio](/es/desktop-ipc-transport/).
Para sesiones entre procesos o entre mﾃ｡quinas, consulta [Transporte Socket](/es/socket-transport/).
Para harnesses locales y pruebas, tambiﾃｩn puedes ejecutar host + remoto en un solo runtime mediante [Transporte en memoria](/es/memory-transport/) (solo dev/test).
Para una guﾃｭa comparativa lado a lado, consulta [Comparaciﾃｳn de transportes](/es/transport-comparison/).

## 4. Validar fronteras

Antes de seguir ampliando, verifica:

- que el host controla la allowlist de componentes expuestos;
- que las props/eventos que cruzan runtimes son serializables;
- que el runtime remoto no depende de APIs directas del DOM del navegador.
